**Time:** 30 minutes  •  **Roles:** admin (+ employee self-view)  •  **Modules:** `/employees`, `/employees/new`, `/employees/[id]`, `/employees/[id]/edit`, `/departments`, `/audit-logs`

## Triage 13May26

Findings grouped following the phase-13 A–F convention (A = correctness, B = exposure, C = UX consistency, D = product enrichments, E = acknowledged non-issues). Batches ordered by risk and dependency: correctness bugs first, then label/exposure, then consistency, then enrichments, then layout.

### Group A — Correctness bugs (data, state, or feedback broken)

- **A1. (Batch 1) Terminate save: edit form reverts Status to Active after a successful save — FIXED (Session 106, 14May26; final in-place fix Session 116, 16May26).** Root cause across the follow-ups: persistence was correct (`employee_records.employment_status` and `end_date` updated), but the edit form's in-place post-save render mixed preserved Server Action state ("Employee updated.") with a native form reset that pushed the Status `<select>` back to its initial Active option. Claude Code could not fully fix this in three tries: first it added edit-route revalidation and controlled fields, which fixed reloads but not the in-place post-save render; then it removed the stale `employee` prop resync, which still let Status revert; then it switched Status/End date to uncontrolled DOM `defaultValue`, which made the reset hazard worse because React form actions reset uncontrolled fields after success. Codex first tested removing `/employees/[id]/edit` self-revalidation; that alone still failed the A1 in-place assertion (`Received: "active"`). Final fix: keep Status + End date controlled from one local client owner, return canonical saved values from `updateEmployee`, remove edit-route self-revalidation, and have `EditEmployeeForm` prevent the default native form-action submit and dispatch the same `useActionState` action with `startTransition`. That preserves the form after save and prevents React's automatic post-success form reset from lying to the admin. End date still auto-defaults to today on Terminated and auto-clears on Active. Pinned by Playwright `A1 — terminate-save persists Status on edit and profile pages` with in-place assertions between toast and reload (`tests/e2e/admin.spec.ts`).
  - On Status = Terminated, default the End date to today (overridable in the same form).
  - On Status flipped back to Active, auto-clear the End date.
  - State owner unchanged: `employee_records.employment_status` + `end_date`. Severity: high (admins lose confidence in saves).
- **A2. (Batch 1) Operational report "Leavers, last 30 days" stays at 0 after a real termination — FIXED (Session 106, 14May26).** Root cause: the Leavers count predicate in `getAdminDashboardData` used `.gte("end_date", sinceDate)`, which silently excludes rows where `end_date IS NULL` — exactly the state created by a termination without a manually-entered end date. Fix: replaced with an OR predicate that also catches null-end_date rows whose `updated_at` is in window (`src/server/dal/dashboard.ts:138-140`). A1's auto-default of End date to today prevents the null-end_date gap on future terminations; the OR fallback back-covers legacy rows. Pinned by Playwright `A2 — Leavers DAL predicate counts null end_date inside the 30d window only` (`tests/e2e/admin.spec.ts:1306`). Clicking the card lands on `/employees?status=terminated` which **does** show the terminated row. So the directory filter works; only the count projection is wrong. Likely a DAL bug in `getAdminDashboardData()` — either filtering on the wrong date column (e.g. `end_date` vs `updated_at`) or excluding the terminated row from the count via the wrong predicate. State owner: `employee_records`. Severity: medium (operational report misleads the admin).
- **A3. (Batch 1) Audit-log search by UUID returns 0 rows for the terminated employee — FIXED (Session 106, 14May26).** Confirmed cause: the only UUID input on `/audit-logs` searched by `actor` (who performed the action), not by `entity_id` (the target record). The DAL (`getAuditLogs`) already supported the `entityId` filter — only the UI was missing. Decision (user selection): added a separate "Entity ID" field rather than a combined field with a toggle. Fix: `src/app/(app)/audit-logs/page.tsx` — added `entityId` to `searchParams` + filters, separate Input between Actor ID and Action, amber "Entity ID filter ignored" banner for invalid UUIDs, clarified Actor ID placeholder. Pinned by Playwright `A3 — audit logs Entity ID filter narrows results and rejects invalid UUIDs` (`tests/e2e/admin.spec.ts:1407`).
