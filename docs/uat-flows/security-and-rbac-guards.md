# Security And RBAC Guards

**Time:** 25 minutes  •  **Roles:** admin → manager → employee → unauthenticated  •  **Modules:** every page, the audit log

A single rotation pass through the negative paths — every place a role is supposed to be blocked. Each guard violation should render the Access Denied page (URL preserved, in place) and emit an `auth.access_denied` audit row with the attempted-resource metadata.

This flow is derived from `userflow.doc` § Task 1.5 ("Negative / Security Flows"). Running it once per rotation is the cheapest way to spot an RLS or `requireRole` regression — every other flow tests one guard in passing; this flow tests all of them in sequence.

## Automation status

Steps marked **[AUTOMATED]** in the tables below are covered by the Playwright suite at [tests/e2e/security-rbac-guards.spec.ts](../../tests/e2e/security-rbac-guards.spec.ts). Run with:

```
npx playwright test --project=security
```

The suite uses the same Forge methodology this doc documents — it captures a legitimate Server Action POST via Playwright route interception (so React-controlled inputs cannot silently revert the swap), substitutes one UUID in the multipart body, and replays from the same authenticated context. See [tests/e2e/forge.ts](../../tests/e2e/forge.ts) for the helper. Pass criteria checked by the harness: deny string in the response body, the victim record is unchanged in the database, and an `auth.access_denied` audit row exists with the victim UUID as `entity_id` plus the expected `metadata.reason`.

Steps **not** marked [AUTOMATED] remain manual. Common reasons: cross-actor capture (the step's action cannot be captured legitimately by the test actor — e.g. step 15), or the action does not actually accept the field the UAT step proposes swapping (step 14 — `submitChangeRequest` always uses `user.id` as the target employee; there is no `employeeId` field in the multipart to forge). Manual steps that share a shape with an automated one (e.g. 26 ≈ 22, 27 ≈ 25) can be added to the suite by copying the existing test — see the coverage matrix comment at the top of the spec file.

When a step has [AUTOMATED] you can still run the manual procedure as a spot-check on the harness; it just isn't required for every rotation.

## Preconditions

- All four seed users sign in. **Alice IS Morgan's direct report.** **Bob is NOT** Morgan's direct report (Operations dept, no manager). This split is what lets Group B test both in-scope and out-of-scope manager paths.
- **Use two separate browser apps (e.g. Chrome and Firefox) or two distinct browser profiles** for the multi-role parts — *not* two tabs in the same profile. The reason is the cross-tab cookie-jar gotcha covered in Pre-Step 0 below.
- Note the latest audit timestamp at `/audit-logs` before starting.
- Have DevTools ready — several steps require submitting a forged payload to a Server Action. The UI "hide the option" is **not** the security boundary; the Server Action is. A step that only verifies the UI is disabled is not green until you also confirm a crafted submission is rejected. Use the **forge methodology** below — *not* in-DOM hidden-input edits.

## Forge methodology (for all Server Action bypass steps)

**Do not** edit hidden `<input>` values in the DOM and click submit. The forms in this app use React-controlled inputs (`value={…}`, not `defaultValue`); React reconciles the DOM attribute back to the component's prop at submit time, so your edit silently reverts and you end up submitting the *original* UUID. That produces a false negative: it looks like the server accepted a forged payload when in reality the server only ever saw the legitimate one. (Session 122 / A12 — Alice's edit reverted to her own `leaveRequestId`, her own leave got cancelled, no `auth.access_denied` was written, and it looked like an RLS bypass until we checked the audit row's `entity_id`.)

Use this recipe instead — it bypasses React entirely by replaying the request at the network layer:

1. **Capture** — perform the action *legitimately* once to record the wire format.
   - Sign in as the test actor (e.g. Alice).
   - Open DevTools → Network tab, filter to `Fetch/XHR`, tick **Preserve log**.
   - Trigger the action on your own record (e.g. submit/cancel your own leave, complete your own onboarding task). This produces a `POST` to the page URL with a `Next-Action` request header — that's the Server Action call.
2. **Copy** — right-click that `POST` → **Copy** → **Copy as fetch** (Chrome/Edge) or **Copy → Copy as fetch** (Firefox). Paste into the DevTools **Console**.
3. **Forge** — in the pasted snippet, find the multipart body. Each form field appears as a `name="…"` block followed by its value. Replace the target field value with the victim's UUID (e.g. swap your own `requestId` / `taskId` / `employeeId` for Bob's). Leave every other field, header, and the boundary string untouched.
4. **Fire** — press Enter to run the fetch. Cookies attach automatically (same origin), so the request is authenticated as the current session.
5. **Verify** — three things must all be true for the step to pass:
   - Response status is non-2xx **or** the response body contains an error message ("Access denied", "You can only…", "Not found"). A response that renders the Access Denied page (URL preserved) also counts — after B4, role denials throw an in-place error instead of redirecting to `/access-denied`.
   - The victim's record is **unchanged** in the database (check via `/audit-logs` as admin, or the relevant module page).
   - A new `auth.access_denied` audit row exists with `actor = <test actor>`, the right `entity_id`, and `metadata` naming the reason. **No audit row = a finding** (observability gap), even if the rejection itself was correct.

Diagnostic: if you're not sure whether the server saw your forged UUID at all, check the audit row's `entity_id`. If it equals *your own* UUID, the forge didn't land — re-do step 3. If it equals the *victim's* UUID and the action is `auth.access_denied`, the guard worked. If it equals the victim's UUID and the action is the success action (`leave.cancelled`, `task.completed`, etc.), that's a real bypass — stop and file critical.

Anywhere a step below says "forge a `X` submission with …", apply this recipe with the named field set to the named victim UUID.

## Audit log events to verify

This flow is essentially an audit-log generator. Expect:

- `auth.access_denied` × many (one per failed access attempt with server-side enforcement).
- No other audit events should fire (this is a read-only / denial-only pass).
- Every `auth.access_denied` row must include `attempted_resource` metadata.

If a step appears to succeed (no `/access-denied`, no audit row), **stop the rotation and file the finding immediately** — that's a guard regression.

## Notes for the reviewer

The four highest-impact regressions to look for in this flow:

1. **Any step in Group A 1–9 succeeding for Alice** — that's a `requireRole` or RLS gap on an admin/manager-only surface.
2. **Any forged-form step in Group A 11–15 succeeding** — that's a Server Action guard regression on the most sensitive write paths (onboarding completion, leave cancel, document upload, change request, manager-score). Phase 9 had real bugs here; this is where they would come back.
3. **Step 10 (Bob's profile by Alice) or step 30 (Alice's profile by Bob) leaking more than the limited People Directory fields** — that's a Session 96 / migration 0033 regression, treat as critical.
4. **Step 21 (Morgan seeing salary/bank/tax/national_id on Alice's profile)** — that's the headline manager-cannot-see-payroll rule breaking; critical.

The Manager / Admin cannot-self-action steps (22, 25, 26, 27) are the most subtle — they require a crafted submission to test the server-side guard, since the UI usually hides the option. If the UI's "hide the option" was the only defence, the server will accept the submission and you'll find a real bug. Use the [Forge methodology](#forge-methodology-for-all-server-action-bypass-steps) (network-layer replay) — *not* in-DOM hidden-input edits — before declaring those steps green.

This flow pairs well with `leave-request-lifecycle.md` and `employee-profile-lifecycle.md` — those exercise the happy paths, this one exercises the guard rails.
